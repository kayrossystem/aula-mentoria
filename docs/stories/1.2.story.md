## Contexto

Durante a migração para v2.1, mais de 20 testes foram desabilitados (via `xit`, `xtest`, `xdescribe`, `.skip`) e nunca reativados. A cobertura real do código core está abaixo de 40%.

Isso significa que qualquer mudança no core pode introduzir regressões que chegam ao usuário final sem alerta. Com type safety agora ativo (Story 1.1), este é o segundo pilar da rede de segurança do sistema.

**Depende de:** Story 1.1 (checkJs ativo) — reativar os testes com tipos quebrados geraria ruído adicional.

---

## Acceptance Criteria

- [ ] **AC1:** Zero ocorrências de `xit(`, `xtest(`, `xdescribe(`, `.skip(` nos arquivos de teste
- [ ] **AC2:** Cobertura global de testes >= 60% (medida via `npm run test:coverage`)
- [ ] **AC3:** Todos os testes reativados passam (verde) ou têm bug correspondente documentado em issue

## Notas para @dev (Dex)

**Estratégia de triagem (ordem de prioridade):**

1. Testes que falharam porque a **implementação mudou**: atualize o teste para refletir o comportamento correto atual (não o comportamento antigo).
2. Testes que falham por **bug real no código**: não corrija o bug nesta story. Crie a issue, adicione `// TODO: issue #NNN` e mantenha o teste desabilitado APENAS se o bug for crítico (com justificativa no comentário).
3. Testes que são **flaky** (falham intermitentemente): investigue a causa raiz antes de reativar. Não aceite flakiness como normal.

**Meta de cobertura por camada:**
- `bin/` (CLI): >= 50%
- `packages/core/` (orchestrator): >= 65%
- `packages/*/src/` (libraries): >= 60%

**Comando para ver cobertura por arquivo:**
```bash
npm run test:coverage -- --coverage-reporter=text | head -80
```
