## Contexto

`master-orchestrator.js` tem 54KB — aproximadamente 1.500-2.000 linhas com responsabilidades múltiplas concentradas em um único arquivo. Isso cria:

- Gargalo de revisão: toda mudança de orquestração toca o mesmo arquivo (conflitos de merge frequentes)
- Testabilidade próxima de zero sem refatoração (impossível mockar dependências)
- Bus factor alto: apenas quem conhece o arquivo inteiro faz mudanças seguras
- Onboarding caro: novo dev precisa entender 2.000 linhas antes de contribuir

Esta story executa a decomposição: primeiro mapeia as responsabilidades, depois extrai módulos.

**Gate de entrada:** EPIC-001 concluído — type safety ativo e cobertura >= 60%.

## Notas para @dev (Dex) e @architect (Aria)

**Princípio guia:** Não invente arquitetura nova — extraia o que já existe implicitamente no código. O diagrama do T2 é a etapa mais crítica: identifique responsabilidades reais, não ideais.

**Sinais de uma boa decomposição:**
- Cada módulo pode ser testado sem carregar os outros
- Cada módulo tem um único "motivo para mudar"
- O facade/index tem menos de 50 linhas

**Sinais de uma decomposição ruim:**
- Módulos que importam uns aos outros em ciclo
- Um módulo que ficou com 80% da lógica original
- Interfaces entre módulos mais complexas que o código original

**Estratégia de extração segura:**
1. Copie o código para o novo módulo
2. Adicione testes para o novo módulo
3. Faça o módulo original importar e reexportar do novo
4. Rode os testes — devem passar sem mudança
5. Só então remova o código duplicado do original
